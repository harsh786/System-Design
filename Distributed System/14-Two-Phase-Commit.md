# Two-Phase Commit (2PC) and Three-Phase Commit (3PC)

## 1. Problem Statement: Atomic Commitment

In a distributed system, a single logical transaction may span multiple nodes (databases, services, shards). The **atomic commitment problem** requires:

- **All-or-nothing**: Either ALL participants commit, or ALL abort
- **Agreement**: No two participants reach different decisions
- **Validity**: If all vote YES and no failures, the decision is COMMIT
- **Termination**: Every non-failed participant eventually decides


### Phase 1: Prepare (Voting Phase)

The coordinator asks all participants if they are able to commit.

**Coordinator actions:**
1. Write `PREPARE` record to its WAL (Write-Ahead Log)
2. Send `PREPARE` message to all participants
3. Wait for responses (with timeout)

**Participant actions on receiving PREPARE:**
1. Determine if it CAN commit (check constraints, acquire locks)
2. If YES: Write all transaction data + `VOTE-YES` to WAL, flush to disk
3. If NO: Write `VOTE-NO` to WAL, abort locally, release locks
4. Send vote back to coordinator

**Critical invariant**: Once a participant votes YES, it has made a **promise** that it CAN commit. It has persisted everything needed to commit later, even after a crash. It CANNOT unilaterally abort after voting YES.

### Phase 2: Commit/Abort (Decision Phase)

The coordinator makes the final decision and broadcasts it.

**Decision rule:**
- If ALL participants voted YES вҶ’ decision is COMMIT
- If ANY participant voted NO (or timed out) вҶ’ decision is ABORT

**Coordinator actions:**
1. Write decision (`COMMIT` or `ABORT`) to WAL вҖ” **this is the commit point**
2. Send decision to all participants
3. Wait for acknowledgments
4. Write `END` record to WAL after all ACKs received

**Participant actions on receiving decision:**
- COMMIT: Apply changes, release locks, write `COMMIT` to WAL, send ACK
- ABORT: Rollback changes, release locks, write `ABORT` to WAL, send ACK

## 3. The Blocking Problem

This is 2PC's **fatal flaw** and the primary motivation for 3PC.

### The Scenario

After a participant votes YES, it enters the **PREPARED** (uncertain) state. It has:
- Written all data to WAL (can commit)
- Acquired all locks (blocking other transactions)
- Made a promise it can commit

But it does NOT know the coordinator's decision. If the coordinator crashes:

## 5. 2PC Optimizations

### Presumed Abort

**Insight**: If the coordinator crashes and has no record of a transaction, it can safely tell any recovering participant to ABORT.

```

- No force-write for ABORT decision
- No ACKs needed for ABORT
- Recovering participant asks вҶ’ no record found вҶ’ ABORT

Abort path: 2 fewer forced disk writes per transaction
```

### Presumed Commit

**Insight**: Optimization for commit-heavy workloads. Don't log ACKs for commits.

```
Presumed Commit logging:
в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ
Coord: log PREPARE + participant list (force)
Coord: log ABORT (force) вҖ” only if aborting
       (don't log COMMIT вҖ” it's presumed)
Coord: log END вҖ” when all ACKs collected

If coord crashes after decision:
- No ABORT record found вҶ’ COMMIT is presumed
- Participants who haven't heard: commit
```

### 2PC vs 3PC Comparison

| Property | 2PC | 3PC |
|----------|-----|-----|
| Message rounds | 2 | 3 |
| Blocking on coord failure | YES | No (synchronous model) |
| Safe under partition | YES (blocks, but consistent) | NO (can be inconsistent) |
| Used in practice | Widely | Almost never |
| Latency | 2 RTTs | 3 RTTs |
| Complexity | Moderate | High |

**Industry consensus**: 2PC + Paxos-replicated coordinator log is preferred over 3PC. You get non-blocking (via coordinator failover) AND safety under partitions.

## 8. Real-World Implementations

### MySQL XA Transactions

```sql
-- MySQL XA: External distributed transactions
XA START 'xid-transfer-001';
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
XA END 'xid-transfer-001';
XA PREPARE 'xid-transfer-001';
-- ... coordinator collects votes from all shards ...
XA COMMIT 'xid-transfer-001';

-- Recovery: find in-doubt transactions
XA RECOVER;
-- Returns list of prepared but not committed XIDs

-- Internal XA (binary log + InnoDB):
-- MySQL uses internal 2PC to keep binlog and InnoDB in sync
-- binlog acts as coordinator, InnoDB as participant
```

**MySQL XA Limitations:**
- XA transactions are not safe with replication (pre-8.0)
- `XA RECOVER` may show stale transactions after crash
- No automatic resolution of in-doubt transactions
- MySQL 8.0 improved with atomic DDL and binlog-based recovery

### PostgreSQL PREPARE TRANSACTION

```sql
-- PostgreSQL: two-phase commit for distributed systems
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
PREPARE TRANSACTION 'transfer-001-node1';
-- Transaction is now "prepared" and persistent
-- Survives server restart!

-- Later (possibly after coordinator decision):
COMMIT PREPARED 'transfer-001-node1';
-- or
ROLLBACK PREPARED 'transfer-001-node1';

-- Find prepared transactions:
SELECT * FROM pg_prepared_xacts;
-- gid | owner | database | prepared_at
-- Shows all in-doubt transactions

-- Configuration:
-- max_prepared_transactions = 100 (default 0 = disabled!)
```

### Microsoft MSDTC

```
Windows Distributed Transaction Coordinator:
- Built into Windows OS (COM+ / .NET)
- Supports XA, OLE Transactions, WS-AtomicTransaction
- Coordinates across SQL Server, MSMQ, Oracle, etc.
- Used heavily in enterprise .NET applications
- Configuration: Component Services вҶ’ My Computer вҶ’ MSDTC

Typical .NET usage:
using (var scope = new TransactionScope())
{
    // Operations on SQL Server A
    // Operations on SQL Server B
    // Operations on MSMQ
    scope.Complete(); // Triggers 2PC
}
// Dispose without Complete() вҶ’ Abort
```

### Java XA Managers (Atomikos / Narayana)

```
Atomikos (embedded TM):
- Runs in-process with the application
- No external coordinator process needed
- Supports JTA/XA for JDBC + JMS
- Automatic crash recovery from transaction logs
- Used in Spring Boot with spring-boot-starter-jta-atomikos

Narayana (JBoss/WildFly):
- Full JTA implementation (formerly Arjuna)
- Supports XA, WS-AtomicTransaction, REST-AT
- Integrated into WildFly/JBoss EAP
- Provides tooling for heuristic decisions
- Compensation-based transactions (extended model)
```

### TCC (Try-Confirm-Cancel)

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ               TCC: TRY-CONFIRM-CANCEL                            в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  Phase 1 (Try): Reserve resources, don't commit yet             в”Ӯ
в”Ӯ  Phase 2 (Confirm): Finalize all reservations                   в”Ӯ
в”Ӯ  Phase 2 (Cancel): Release all reservations (on failure)        в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  Example: Flight + Hotel booking                                 в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  TRY:                                                            в”Ӯ
в”Ӯ    Flight service: Reserve seat (status=RESERVED, not BOOKED)   в”Ӯ
в”Ӯ    Hotel service:  Reserve room (status=HELD, not CONFIRMED)    в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  CONFIRM (all tries succeeded):                                  в”Ӯ
в”Ӯ    Flight: RESERVED вҶ’ BOOKED                                    в”Ӯ
в”Ӯ    Hotel:  HELD вҶ’ CONFIRMED                                     в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  CANCEL (any try failed):                                        в”Ӯ
в”Ӯ    Flight: RESERVED вҶ’ AVAILABLE (release seat)                  в”Ӯ
в”Ӯ    Hotel:  HELD вҶ’ AVAILABLE (release room)                      в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  vs 2PC: Resources are "soft-locked" (reserved) not hard-locked в”Ӯ
в”Ӯ  Other users can see "2 seats remaining" (not blocked)          в”Ӯ
в”Ӯ  Reservation timeout provides automatic cancel                   в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### Google Percolator (Optimistic Transactions)

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ            PERCOLATOR: OPTIMISTIC 2PC ON BIGTABLE                в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  No locks during execution. Conflict detection at commit time.  в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  Write flow:                                                     в”Ӯ
в”Ӯ  1. Read data, note timestamps                                  в”Ӯ
в”Ӯ  2. Buffer writes locally                                       в”Ӯ
в”Ӯ  3. PreWrite: Write tentative values with locks (to Bigtable)   в”Ӯ
в”Ӯ     - Pick one key as "primary lock"                            в”Ӯ
в”Ӯ     - Other keys reference primary (secondary locks)            в”Ӯ
в”Ӯ  4. Commit primary: if primary lock still held вҶ’ committed      в”Ӯ
в”Ӯ     - Write commit timestamp, remove lock                       в”Ӯ
в”Ӯ  5. Async: clean up secondary locks                             в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  Conflict: If lock exists on a key вҶ’ another txn writing        в”Ӯ
в”Ӯ    вҶ’ Check if that txn's primary is still locked                в”Ӯ
в”Ӯ    вҶ’ If expired вҶ’ roll it back and proceed                      в”Ӯ
в”Ӯ    вҶ’ If fresh вҶ’ back off and retry                              в”Ӯ
в”Ӯ                                                                 в”Ӯ
в”Ӯ  Used by: TiDB (TiKV), CockroachDB (inspired by)               в”Ӯ
в”Ӯ  Key insight: Decentralized вҖ” no dedicated coordinator node     в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### The Cost Matrix

| Approach | Consistency | Availability | Latency | Complexity | Lock Duration |
|----------|------------|--------------|---------|------------|---------------|
| 2PC/XA | Strong | Low (blocking) | 2-3 RTT | Moderate | Entire protocol |
| 3PC | Strong* | Higher | 3-4 RTT | High | Entire protocol |
| Saga | Eventual | High | Per-step | High (compensations) | Per-step only |
| TCC | Reservation | High | 2 phases | Moderate | Soft (timeout) |
| Outbox | Eventual | High | Async | Low | None cross-svc |
| Paxos+2PC | Strong | High (non-blocking) | Consensus + 2PC | Very high | Short (optimized) |

*3PC strong consistency only holds in synchronous networks (unrealistic)
